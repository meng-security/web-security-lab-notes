# SQLi-Labs Less-1 联合查询型 SQL 注入漏洞复现报告

## 1. 漏洞概述
本次测试在自建的 SQLi_labs 授权靶场中进行。
测试发现 Less-1 页面中的 id 参数存在 SQL 注入漏洞。

攻击者可通过构造恶意输入影响后端 SQL 查询逻辑。在页面存在数据回显的条件下，可进一步使用联合查询读取数据库名称、表名、字段名以及靶场中的模拟账户数据。

漏洞类型：联合查询型 SQL 注入
请求方式：GET
参数位置：id 参数
风险等级：高风险（仅针对授权靶场的演示环境）

## 2. 测试授权与范围
本次测试仅在个人搭建的隔离实验环境中进行，目标为 SQLi-Labs 漏洞练习靶场，不涉及任何真实业务系统、真实用户数据或未授权资产。

测试范围：
http://192.168.0.111/Less-1/?id=1

## 3. 实验环境
虚拟化平台：VMware Workstation

攻击机：
操作系统：Kali Linux
IP 地址：192.168.0.104
工具：Firefox、Burp Suite Community Edition

靶机：
操作系统：Windows 10
IP 地址：192.168.0.111
靶场：SQLi-Labs Less-1
应用运行环境：Apache + PHP
数据库：MySQL

网络模式：
攻击机与靶机均使用 VMware 桥接模式。

## 4. 测试思路与方法
本次测试先从正常访问页面开始，确认 id 参数能够正常返回用户信息。
然后在参数中加入单引号等异常字符，观察页面是否出现报错，从而判断该参数是否可能存在 SQL 注入。
发现报错后，再通过构造“条件成立”和“条件不成立”的请求进行对比，确认输入是否能够影响后端查询，并判断参数大概是什么闭合方式。
由于页面能够直接显示查询结果，所以选择使用联合查询的方式继续验证。后续通过判断字段数量、确认回显位置，再读取数据库中的表名、字段名和靶场模拟数据，验证漏洞可能造成的信息泄露风险。
最后结合漏洞产生原因，给出对应的修复建议和复测思路。

## 5. 漏洞复现过程
### 5.1 正常功能基线验证
测试目的：确认 SQLi-Labs Less-1 页面能够正常访问，并记录 id 参数正常取值时的页面返回内容，作为后续漏洞测试的对照。

测试请求：使用 Burp Suite Repeater 对以下请求进行重放：
GET /Less-1/?id=1 HTTP/1.1
Host: 192.168.0.111

响应现象：服务端正常返回 200 OK，页面显示用户信息，包括用户名 Dumb 和密码 Dumb。

分析与结论：当 id 参数值为 1 时，页面会返回对应的用户信息，说明该参数会影响页面显示的数据，推测其可能参与了后端数据库查询。后续将通过修改该参数，验证是否存在 SQL 注入漏洞。

证据截图：图 2 正常 id 参数请求及响应结果
### 5.2 异常输入与注入点初步确认
测试目的：
通过向 id 参数中加入异常字符，观察服务端响应是否出现数据库错误信息，初步判断该参数是否存在 SQL 注入风险。

测试请求：
使用 Burp Suite Repeater 将 id 参数由正常值 1 修改为：
1'

响应现象：
服务端返回 HTTP 200 状态码，但页面响应中出现 MySQL SQL 语法错误信息。

分析与结论：
单引号字符导致后端 SQL 查询语句执行异常，说明 id 参数中的用户输入可能未经有效过滤或参数化处理，直接参与了数据库查询。该参数存在 SQL 注入风险，需要继续判断具体的注入类型、闭合方式及可利用程度。

证据截图：
图 3 异常单引号输入触发 SQL 语法报错
### 5.3 注入闭合方式及可控性验证
测试目的：
验证 id 参数所在的 SQL 语句上下文，判断闭合方式和注释方式，并确认输入是否可控制查询逻辑。

测试请求：
逻辑真条件：
id=1' AND '1'='1' -- 

逻辑假条件：
id=1' AND '1'='2' -- 

响应现象：
真条件请求仍返回 Dumb 用户信息；假条件请求不再返回该用户信息，且两次请求均未出现 SQL 语法错误。

分析与结论：
请求中的单引号能够成功闭合原始 SQL 语句中的字符串条件，-- 能够注释原始查询后续内容。真、假条件造成不同响应，说明输入可影响后端 SQL 查询逻辑。由此确认该参数为单引号闭合的字符型 SQL 注入。

证据截图：
图 4 逻辑真条件请求结果。
图 5 逻辑假条件请求结果。
### 5.4 回显情况分析与联合注入方式选择
测试目的：
根据前面测试的响应结果，判断页面是否存在数据回显位置，并选择合适的注入验证方式。

前序测试结果：
正常请求及逻辑真条件请求中，页面均会返回用户名和密码等用户信息；异常输入时，页面还会返回数据库语法报错信息。

分析与结论：
目标页面能够稳定显示数据库查询结果，说明页面存在明显的数据回显位置。攻击者可以尝试通过 UNION SELECT 将额外查询结果拼接到原始查询中，并显示在页面上。

因此，本次漏洞验证选择联合查询注入方式。后续将先确认原始 SQL 查询的字段数量，再确定哪些字段能够在页面中回显，最后在授权靶场内验证可读取的数据库信息。

证据说明：
本节结论基于图 2 至图 5 的正常响应、真假条件响应及报错信息得出。
### 5.5 原始查询字段数判断
测试目的：
确认原始 SQL 查询的字段数量，为后续联合查询构造匹配的 UNION SELECT 语句。

测试请求：
依次将 id 参数修改为：
1' ORDER BY 1 -- 
1' ORDER BY 2 -- 
1' ORDER BY 3 -- 
1' ORDER BY 4 -- 

响应现象：
当使用 ORDER BY 3 时，页面正常返回用户信息；当使用 ORDER BY 4 时，页面出现 SQL 语法错误。可判断原始查询结果共有 3 列。

分析与结论：
ORDER BY 用于按查询结果中的字段位置排序。当排序字段位置未超过原始查询字段数量时，SQL 可以正常执行；当字段位置超过实际字段数量时，数据库会返回错误。根据测试结果判断，原始 SQL 查询结果共有 3 列。

证据截图：

### 5.6 联合查询回显位确认
测试目的：
确认联合查询结果中哪些字段能够显示在页面中，为后续验证数据库信息提供可用的回显位置。

测试请求：
id=-1' UNION SELECT 1,2,3-- 

响应现象：
页面未返回原始的 Dumb 用户信息，而是显示联合查询构造的数据。其中，数字 2 显示在用户名位置，数字 3 显示在密码位置。

分析与结论：
通过联合查询测试可知，原始查询共有 3 列，其中第 2 列和第 3 列能够在页面中直接回显。后续可将数据库查询结果放入这两个回显位置，验证漏洞造成的信息泄露影响。

## 6. 漏洞危害分析
### 6.1 已验证风险
本次测试已验证，攻击者可利用 id 参数中的 SQL 注入漏洞，通过联合查询获取数据库相关信息，包括当前数据库名称、MySQL 版本、数据表名称、字段名称以及 SQLi-Labs 靶场中的模拟账户数据。

该漏洞证明攻击者能够绕过应用原本的数据查询逻辑，读取数据库中的非预期内容。在真实业务场景中，若存在类似漏洞，可能导致用户信息、订单信息、业务配置等敏感数据泄露。
### 6.2 潜在风险
本次测试仅在个人搭建的 SQLi-Labs 授权靶场中验证了信息读取风险，未对数据修改、删除、权限绕过等破坏性操作进行测试。

若真实业务系统中的数据库连接账户权限过高，攻击者可能进一步尝试修改或删除数据库数据；若应用认证逻辑与数据库查询存在缺陷，也可能扩大为身份认证绕过等风险。具体影响需结合实际业务逻辑、数据库账户权限及安全防护措施进一步评估。

## 7. 漏洞成因分析
漏洞产生的根本原因是后端程序将用户可控的 id 参数直接拼接进 SQL 查询语句，并且未对输入进行有效的参数化处理。

攻击者输入的单引号可以闭合原有 SQL 字符串，再通过 UNION SELECT 等语句拼接额外查询内容，从而影响原本的数据库查询逻辑。

此外，应用将详细的 MySQL 语法错误直接返回给用户，也会帮助攻击者更快判断数据库类型、语句结构和注入方式。

## 8. 修复建议
1. 使用预编译语句或参数化查询
后端数据库操作应使用 Prepared Statement，将用户输入作为参数传入，而不是直接拼接 SQL 字符串。

2. 对输入参数进行类型校验
id 参数应限制为整数类型，例如仅允许数字输入；对于不符合格式的请求应直接拒绝。

3. 关闭详细数据库报错
生产环境不应向用户返回 MySQL 语法错误、SQL 语句片段等敏感调试信息。详细错误应只记录在服务端日志中。

4. 最小权限配置数据库账户
Web 应用连接数据库时，应使用权限最小化的账户，避免使用 root 或具备高权限的数据库账号。

5. 增加安全监控与防护
可结合 WAF、日志监控、异常请求检测等措施，识别 UNION SELECT、information_schema 等高风险请求特征。

## 9. 复测建议
修复完成后，应重新对 id 参数进行单引号、逻辑真/假条件及联合查询测试。

预期结果包括：
1. 输入单引号不会出现数据库语法错误；
2. 真、假条件不会影响页面返回数据；
3. UNION SELECT 等恶意输入无法获取数据库信息；
4. 服务端仅返回统一的错误提示，不暴露 SQL 细节。

## 10. 总结
本次在自建 SQLi-Labs 授权靶场中，使用 Kali Linux 和 Burp Suite 对 Less-1 页面 id 参数进行了 SQL 注入漏洞验证。

测试按照“正常功能验证、异常输入测试、闭合方式判断、逻辑真/假条件验证、联合查询字段数判断、回显位确认、数据库信息枚举、模拟账户数据读取”的流程进行。

最终确认该参数存在单引号闭合的字符型 SQL 注入漏洞，并且由于页面存在数据回显位置，可被联合查询利用，导致数据库结构和靶场模拟账户数据泄露。
